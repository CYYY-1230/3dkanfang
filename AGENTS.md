# 3d-house-viewer 项目规则

## 项目定位

- 项目名称：`3d-house-viewer`
- 项目简介：这是一个面向普通买房用户的网页端 3D/VR 看房系统。
- 核心数据方案：地图 POI + 户型数据。
- 主要目标：让普通用户能清楚、轻松地了解房源位置、周边配套、户型空间和 3D/VR 看房体验。

## 技术方向

- 使用 Next.js 作为网页应用框架。
- 使用 React 构建页面和交互组件。
- 使用 TypeScript 编写代码，减少低级错误。
- 使用 Tailwind CSS 管理页面样式。
- 使用 Prisma 管理数据库访问。
- 使用 PostgreSQL 作为数据库。
- 地图 SDK 后续再确定。
- VR 查看器后续再确定。
- 邮件服务后续再确定。
- 部署平台后续再确定。

说明：

- Next.js：用于开发 React 网页应用的框架，能同时处理页面、路由和服务端能力。
- React：用于把页面拆成可复用组件的前端库。
- TypeScript：给 JavaScript 增加类型检查，帮助提前发现错误。
- Tailwind CSS：用工具类快速写样式的 CSS 框架。
- Prisma：让代码更方便、安全地读写数据库的工具。
- PostgreSQL：常用的关系型数据库，用来保存结构化数据。

## UI 风格

- 整体风格要高级、清爽、有房产科技感。
- 不要做成后台管理系统的感觉。
- 不要做成普通电商商品详情页的感觉。
- 移动端必须友好，普通用户在手机上也要容易浏览和操作。
- 页面应优先服务买房用户的真实看房需求，而不是堆砌功能。

## 开发协作规则

- 每个任务先计划，再执行。
- 一个对话框只做一个明确任务。
- 不扩展无关功能。
- 不删除文件，除非用户明确要求。
- 如果需求不清楚，先提出关键问题，不要直接猜着做。
- 如果一个任务有风险，先说明风险，再执行。
- 修改完成后，总结改了哪些内容、如何检查、下一步建议。
- 每次回答尽可能用小白能听懂的话解释。
- 遇到技术名词时，用一句简单的话解释它是什么意思。

## 安全与配置规则

- 不要硬编码 API Key、数据库密码、邮件密钥等敏感信息。
- 使用环境变量管理密钥。
- 如果需要新增环境变量，必须同步说明应该添加什么变量、变量用途是什么。
- 不要把真实密钥写入代码、文档示例或提交记录中。

说明：

- API Key：第三方服务用来识别项目身份的钥匙。
- 环境变量：放在运行环境里的配置，适合保存数据库地址、密钥等不应写死在代码里的内容。
