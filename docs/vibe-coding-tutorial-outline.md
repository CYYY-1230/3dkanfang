# 3D 看房工具从 0 到 1 Vibe Coding 教程大纲

## 教程定位

这套教程面向 AI 自媒体观众，重点不是逐行讲代码，而是展示如何和 Codex 协作，把一个模糊想法拆成清晰需求、可执行流程和可运行产品。

项目最终要呈现的是一个网页端 3D/VR 看房 demo：用户可以在地图上看小区，查看周边配套和户型资料，再进入 3D/VR 看房页面，并体验装修风格切换和收藏。

当前版本属于产品演示版。价格、成交价、户型资料和部分看房素材可以由 AI 生成或本地规则生成，但必须明确标注为示意内容，不是真实购房参考。

## 1. 梳理需求：我们到底要做什么

这一章先把“我要做一个 3D 看房工具”拆成观众能听懂、AI 能执行的需求。

- 目标：做一个网页端 3D/VR 看房工具。
- 用户：普通买房用户，不是房产中介后台，也不是普通电商商品页。
- 核心场景：用户在地图上看小区，点进楼盘，看户型，再进入 3D/VR 看房。
- 核心能力：地图找房、周边配套、户型展示、装修风格切换、VR 看房、收藏。
- 数据思路：地图 POI 提供小区位置，户型和 VR 素材用 AI 生成示意数据。
- 第一版边界：先做 demo，不追求真实房源交易数据。

可以强调一个关键观点：需求不是越多越好，第一版要先找出最短闭环。

最短闭环是：

```text
地图找房 -> 看楼盘 -> 看户型 -> 进入 VR 看房 -> 收藏
```

## 2. 梳理完整工作流程

这一章提前给观众一张完整路线图，让他们知道后面每一步为什么要这样做。

完整流程：

1. 写项目规则，让 AI 知道技术栈、UI 风格、安全边界和开发习惯。
2. 讨论 MVP，确定第一版只跑通核心看房路径。
3. 先搭前端页面，用模拟数据跑通体验。
4. 优化首页高级感和用户路径。
5. 接入高德地图，让地图从假点位变成真实地图。
6. 用高德 POI 找真实小区，再用 AI/规则生成合理示意资料。
7. 做 3D/VR 看房，用 Three.js 实现真 360 全景。
8. 用 Image Gen 生成户型图、白膜图、风格图，并不断校对。
9. 自查、修 bug、重启服务、验证页面和交互。

这一章要告诉观众：vibe coding 不是一句话生成完整项目，而是持续表达、检查、修正和推进。

## 3. 建立项目规则和产品边界

这一章展示先创建 `AGENTS.md`，让 Codex 在后续开发中有明确边界。

要讲清楚这些规则：

- 项目名称：`3d-house-viewer`。
- 技术栈：Next.js、React、TypeScript、Tailwind CSS、Prisma、PostgreSQL。
- UI 调性：高级、清爽、有房产科技感、移动端友好。
- 产品边界：不要做成后台管理系统，也不要做成普通电商商品页。
- 安全规则：API Key、数据库密码、邮件密钥都走环境变量。
- 协作规则：先计划再执行，一个对话尽量只做一个明确任务。

小白解释：

- API Key：第三方服务给项目用的钥匙。
- 环境变量：放在本地或服务器里的配置，不直接写进代码。
- 技术栈：完成项目所用的一组工具。

这一章的重点是：先写规则，AI 才不会乱选技术、乱加功能、乱写密钥。

## 4. 搭建前端 MVP

这一章展示如何从空项目搭出能演示的前端版本。

第一版先不等数据库，也不急着接真实地图，先用模拟数据跑通完整体验。

页面包括：

- 首页：展示找房入口和地图找房区域。
- 楼盘详情页：展示楼盘介绍、地址、周边配套、可选户型。
- 户型详情页：展示户型图、面积、朝向、亮点和进入 VR 的按钮。
- VR 看房页：展示房间切换、风格切换和沉浸式看房区域。
- 收藏页：让用户看到自己保存的户型。
- 登录页：先做简单体验，不急着接复杂认证。

这一章要强调：MVP 的目标不是完美，而是让观众看到产品链路已经能跑起来。

## 5. 打磨首页和核心体验

这一章讲真实迭代过程：先做出来，再变好看，再变好用。

最开始的首页可能像功能原型，只是地图和列表摆在一起。后面要把它打磨成高级找房体验。

优化方向：

- 首屏更像正式产品，而不是工具面板。
- 配色使用暖白石材、深松绿、香槟金。
- 地图、筛选、楼盘列表之间要有联动。
- 用户路径更明确：先选房，再看详情，再看户型，再进 VR。
- 按钮文案要指向下一步，而不是让用户猜。

可以展示一个真实判断：高级感不是多加装饰，而是信息更克制、层级更清楚、下一步更自然。

## 6. 实现地图、AI 示意资料和 VR 看房

这一章合并讲项目里最核心的功能升级：真实地图、真实小区入口、AI 示意资料和 3D/VR 看房。

### 地图能力

接入高德地图时需要准备：

- Web JS API Key：用于网页显示地图。
- Security JS Code：用于高德 JS API 安全校验。
- Web 服务 Key：用于后端查询 POI。

实现方式：

- 前端加载高德地图。
- 地图上展示楼盘和小区点位。
- 后端接口查询周边 POI，避免把 Web 服务 Key 暴露到浏览器。

POI 可以理解为地图上的兴趣点，比如小区、地铁站、学校、商场、医院、公园。

### AI 示意资料

高德可以提供小区名、地址、坐标、区域和周边配套，但不能直接提供完整房产数据。

所以 demo 采用：

```text
高德提供真实小区入口
AI / 本地规则生成示意看房资料
```

示意资料可以包括：

- 示意总价。
- 示意成交价。
- 示意均价。
- 小区简介。
- 适合人群。
- 户型标签。
- 周边配套标签。

页面必须标注：

- AI 示意资料。
- 非真实成交数据。
- 仅用于产品演示。

### 数据量控制

为了避免地图卡顿，不一次性加载全城数据。

控制方式：

- 当前版本只看杭州区域。
- 只查当前楼盘附近或当前地图视野里的小区。
- 低缩放层级只显示小点。
- 放大到小区层级，或点击小区后，再显示完整信息卡。

按需加载的意思是：用户看到哪里，系统才加载哪里的数据。

### VR 看房能力

早期错误路线是把一张图做背景平移，看起来像拖动长图，不是真 VR。

正确路线：

- 使用 Three.js。
- 把 360 全景图贴到球体内壁。
- 相机放在球体中心。
- 鼠标拖拽控制视角。
- 滚轮控制视野大小。
- 点击热点切换房间。
- 切换装修风格时替换当前房间的全景图。

Three.js 可以理解为浏览器里的 3D 图形工具。

### 素材生成路线

素材生成不要一上来就追求完整 3D 建模。

更稳的路线：

1. 先根据户型图整理空间骨架。
2. 每个房间先生成白膜全景图。
3. 白膜图只保留墙、地、顶、门洞、窗洞和基础体块。
4. 再基于白膜图派生装修风格。
5. 每张风格图都要校对门窗、房间关系、机位是否漂移。

风格示例：

- 轻奢意式风。
- 包豪斯风格。
- 原木风格。
- 现代主义风格。
- 中古风格。

这一章的重点是：地图解决位置真实感，AI 解决演示资料完整度，Three.js 解决 VR 交互可信度。

## 7. 自查、踩坑和修复

这一章展示真实开发中如何发现问题、解释问题、修掉问题。

检查项：

- `npm run typecheck`：检查 TypeScript 类型。
- `npm run build`：检查项目能否正式打包。
- 检查地图是否加载成功。
- 检查 POI 是否正常返回。
- 检查环境变量是否被 Next.js 读取。
- 检查 VR 是否是真 360，而不是拖动长图。
- 检查页面在移动端是否能正常浏览。

真实踩坑案例：

- 开发服务运行时又跑 `npm run build`，可能导致 Next.js 的 `.next` 缓存错乱。
- 页面样式像断掉一样，链接变蓝、图片变大。
- 修复方式是重启开发服务，让 Next.js 重新生成开发态资源。

真实校对案例：

- 如果房间门洞、入口或空间关系生成错了，不能只修文案。
- 要重做白膜图和对应风格图。
- 因为白膜是后续所有风格图的空间底稿。

## 8. 结尾：这套方法如何复用

这一章把项目经验收束成可复用方法。

可以总结为：

```text
需求 -> 规则 -> MVP -> 页面 -> 数据 -> 核心交互 -> 自查 -> 迭代
```

复用的不是某几段代码，而是这套工作流。

给观众的核心启发：

- 不要一开始就让 AI 做大而全的系统。
- 先定义边界，再跑通最短闭环。
- 先用模拟数据做体验，再逐步接真实服务。
- AI 生成内容要标注边界，尤其是价格、成交价这类高风险信息。
- 每次改完都要验证，而不是只看 AI 说完成了。

后续可以继续扩展：

- 接真实数据库。
- 接正式用户账号。
- 扩展更多城市。
- 批量生成户型图和全景图。
- 做正式部署。
- 增加真实数据源或合法授权数据。

## 可复用提示词方向

教程最后可以给观众一段可复用提示词：

```text
我想从 0 到 1 做一个产品 demo。请你先不要写代码，先帮我梳理：
1. 目标用户是谁
2. 核心场景是什么
3. 第一版 MVP 只做哪些功能
4. 哪些功能后面再做
5. 需要哪些页面
6. 需要哪些数据
7. 技术实现顺序是什么
8. 有哪些风险和边界

请用小白能听懂的话解释，不要把项目设计得太复杂。
```

然后再进入实现：

```text
请基于刚才确认的 MVP 计划实现第一版。
要求：
1. 先搭可运行前端
2. 先用模拟数据跑通完整路径
3. 不硬编码任何密钥
4. 每次完成后告诉我改了什么、怎么检查、下一步做什么
```
